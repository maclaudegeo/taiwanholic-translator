# 台灣旅遊文章日文翻譯器 Design

**Date:** 2026-07-07

## Summary

建立一個可上傳 `docx` 旅遊文章的網頁工具，將中文內容翻成日文，保留原作者意思，同時把表現層優化成「熟悉台灣、願意帶路的台灣好朋友」語氣。風格參考日本旅遊媒體常見寫法與 Howto Taiwan 的高層次特徵，但不直接複製其可辨識表現。

第一版支援以下內容類型：

- 標題
- SEO 描述
- 小標
- 內文段落
- 圖片說明

第一版輸入與輸出：

- 輸入：`docx`
- 輸出：網頁預覽 + 可下載的新 `docx`

## Product Goals

- 忠實保留原作者原意、資訊與重點順序
- 讓譯文自然、像日本旅遊媒體會寫的日文
- 強化標題吸引力，但不誤導、不添加原文沒有的主張
- 提供嚴格的第二階段日文校正，降低機器翻譯感
- 提供近期日本旅客關注詞建議，但不讓趨勢詞污染正文

## Non-Goals

- 不支援 PDF、HTML、Markdown、純文字批次匯入
- 不做完全模仿或抄襲特定網站的可辨識文風
- 不自動重寫原文立場、刪改內容邏輯、虛構資訊
- 不把趨勢詞自動插入所有段落
- 不提供多品牌語氣切換

## Target Reader And Editorial Voice

### Reader

主要讀者為喜愛台灣的日本旅人，包含：

- 初次訪台，希望資訊安心、清楚、可直接採用的人
- 已多次訪台，希望從觀光進一步走進在地生活的人

### Editorial Voice

品牌核心為「住進台灣日常」。

語氣原則：

- 親切，但不過度撒嬌
- 有陪伴感，像熟悉台灣的朋友在帶路
- 有生活感，不只是景點清單
- 資訊實用、可信、清楚
- 不浮誇，不像硬廣告

### Reference Style Boundary

可參考的高層次特徵：

- 日本旅遊媒體常見的標題包裝方式
- 導語先建立情境，再告訴讀者這篇能解決什麼
- 段落兼顧實用資訊與體驗感

不可做的事：

- 重現 Howto Taiwan 固定句型或可辨識招牌寫法
- 用接近逐句對照的方式模仿文章節奏
- 讓成品看起來像同一媒體直接出稿

## Core Translation Principle

翻譯必須遵守以下優先順序：

1. 內容忠實
2. 日文化表現自然
3. 標題與導語的媒體化優化

### Fidelity Rules

必須保留：

- 作者原本想傳達的意思
- 地名、店名、品牌名、人名
- 價格、交通、營業資訊等事實內容
- 推薦理由與體驗重點
- 原文的主要結構與重點順序

不得任意：

- 擴寫原文沒有的推薦點
- 補上未被提及的背景知識當成正文
- 把保守描述改成絕對語氣
- 把一般敘述硬改成浮誇宣傳

## Content Processing Model

上傳後，系統將文件拆成結構化區塊：

```ts
type BlockType =
  | "title"
  | "seo_description"
  | "heading"
  | "paragraph"
  | "caption";

type ArticleBlock = {
  id: string;
  type: BlockType;
  sourceText: string;
  translatedText: string | null;
  polishedText: string | null;
  trendSuggestions: string[];
  notes: string[];
};
```

所有 AI 流程都以區塊為單位處理，避免整篇失敗時必須全部重跑。

## End-To-End Workflow

1. 使用者上傳 `docx`
2. 後端解析文件內容與順序
3. 系統將內容映射成區塊
4. 第一階段 AI 進行風格化翻譯
5. 第二階段 AI 進行嚴格日文校正
6. 第三階段查詢近期日本旅客關注詞，僅產出可選建議
7. 前端顯示結果，依區塊分類檢視
8. 使用者下載新的 `docx`

## AI Pipeline

### Stage 1: Travel-Style Translation

目的：

- 將中文轉為自然日文
- 保留原意
- 套用品牌語氣
- 對標題與導語進行較積極的表現優化

輸入：

- 區塊類型
- 原文內容
- 品牌語氣規則
- 風格參考摘要

輸出：

- `translatedText`
- 必要時的 `notes`

### Stage 2: Strict Japanese Polishing

目的：

- 抓出不自然直譯
- 修正中文語序殘留
- 檢查日本旅遊媒體會不會這樣寫
- 再次確認標題是否吸引但不誤導

檢查重點：

- 機器翻譯感
- 奇怪搭配詞
- 不像母語者的段落節奏
- 不符合區塊類型的表現方式

輸出：

- `polishedText`
- `notes`

### Stage 3: Trend Suggestion Layer

目的：

- 參考近期日本市場與台灣旅遊相關關鍵字
- 僅針對標題、SEO 描述、小標提供補強建議
- 不直接覆寫正文

套用規則：

- 只建議，不強制
- 以自然性優先
- 不能扭曲原文主題

輸出：

- `trendSuggestions`

## Prompt Strategy

### Shared System Constraints

所有提示詞都必須包含：

- 忠實保留原作者原意
- 禁止虛構資訊
- 以日本讀者自然閱讀感為優先
- 風格參考僅限高層次特徵，不得模仿可辨識文風

### Translation Prompt Requirements

翻譯提示詞至少要明確描述：

- 文章讀者是喜愛台灣的日本旅人
- 品牌定位是「住進台灣日常」
- 語氣要像熟悉台灣、願意帶路的朋友
- 標題可優化，但不可誤導
- 內文重在自然與實用，不浮誇

### Polishing Prompt Requirements

校正提示詞至少要明確描述：

- 你是嚴格的日本編輯與校稿者
- 優先抓不自然日文與翻譯腔
- 如原文意思被改動，必須拉回忠實表達
- 若標題過度誇大，必須收斂

### Trend Prompt Requirements

趨勢詞建議提示詞至少要明確描述：

- 趨勢詞只可作為輔助
- 不得犧牲自然日文
- 不得讓標題變成關鍵字拼裝
- 若無自然可用的詞，可以回傳空建議

## Title Optimization Rules

標題是最可以被優化的區塊，但需遵守以下原則：

- 忠於原主題
- 可加強日本讀者的閱讀動機
- 可參考旅遊媒體慣用結構
- 不得變成與原文無關的 SEO 標題

可接受的優化方向：

- 從直譯改成更自然的名詞句或整理句
- 增加情境感或利益點
- 讓重點更前置

不可接受的優化方向：

- 憑空加入數字
- 憑空加入「必吃」「必買」「完全攻略」等未被支持的強結論
- 為了吸睛而改變文章主軸

## UI Design

第一版頁面包含三個主要區域。

### Left: Upload And Controls

- `docx` 上傳區
- 上傳成功狀態
- 風格設定摘要（固定顯示，不開放大量自訂）
- 趨勢建議開關
- 開始翻譯按鈕
- 下載 `docx` 按鈕

### Center: Translation Results

依序顯示每個區塊：

- 區塊類型
- 原文
- 第一階段翻譯
- 最終校正版

### Right: Editorial Notes

- 校正註記
- 趨勢詞建議
- 標題建議版位
- SEO 描述建議版位

## File Import And Export

### Import

第一版以 `docx` 為唯一正式支援格式。

匯入時需處理：

- 文件段落順序
- 可能存在的標題樣式
- 空白段落過濾
- 圖片說明文字

### Export

輸出新的 `docx` 時需保留：

- 文章區塊順序
- 基本標題層級
- 內文分段
- 圖片說明位置對應

第一版不追求完整保留原始複雜版型，但要確保內容結構清楚可交付。

## Error Handling

- 非 `docx` 檔案：明確提示不支援
- 檔案解析失敗：指出無法解析的原因
- 單一區塊翻譯失敗：保留其他區塊結果
- 模型回應格式異常：回退到安全錯誤訊息
- 趨勢資料抓取失敗：不阻塞主翻譯流程

## Testing Strategy

### Parser Tests

- `docx` 解析為正確區塊
- 標題與段落順序保持一致
- 空段落與雜訊處理正確

### Prompt And Pipeline Tests

- 各區塊被送入正確流程
- 標題走標題優化規則
- 趨勢建議只作用於指定區塊

### Export Tests

- 可成功產生 `docx`
- 區塊順序與內容不丟失
- 校正版文字被正確寫入

### Integration Tests

- 上傳到下載的完整流程可跑通
- 單一階段失敗時前端能顯示合理狀態

## Open Decisions Resolved

- 前端形式：網頁介面
- 輸入格式：`docx`
- 輸出方式：網頁預覽 + `docx` 下載
- 內容範圍：標題、SEO 描述、小標、內文、圖片說明
- AI 流程：翻譯 -> 嚴格校正 -> 趨勢建議
- 品牌語氣：台灣好朋友感
- 風格邊界：參考 Howto Taiwan 高層次特徵，但不做可辨識模仿
- 內容原則：忠實原意優先
